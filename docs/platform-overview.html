duynhlab · internal developer platform

Kiến trúc tổng thể

Dev tự onboard → update env → enable env → promote image qua 4 Backstage template; DevOps/SRE chỉ review PR. Ba Kind cluster: mgmt (Backstage + CNPG) và dev / prod — mỗi môi trường một cluster chạy Flux riêng, sync đúng path của nó trong repo gitops.

01Sơ đồ thành phần & luồng chính

Ba repo GitHub + registry ghcr, ba Kind cluster: mgmt (Backstage + CNPG), dev và prod — mỗi env một Flux riêng. Số trên mỗi cạnh tra ở chú giải bên dưới.

  1. Dev push code lên checkout-service
  2. CI (gha-workflows): test · gitleaks · build · trivy · cosign → push image sha-* bất biến
  3. Flux Image Automation tự bump tag apps/staging (commit thẳng); prod-* qua PR review. Auth bằng GitHub App, không PAT.
  4. Dev dùng portal (guest sign-in)
  5. 4 template (onboard / update-env-var / enable-environment / promote-image) mở PR sang gitops; catalog provider discover catalog/*.yaml chiều ngược lại
  6. DevOps/SRE review + merge (CODEOWNERS + branch protection, 1 review)
  7. Flux trên kind-dev sync clusters/dev mỗi 1 phút (FluxInstance, operator-managed)
  8. Flux trên kind-prod sync clusters/prod — hai env cách ly vật lý, không đọc chéo path
  9. Cluster pull image sha-* và chart mop (OCI) từ ghcr (cả dev lẫn prod)
  10. Backstage đọc workloads + Flux CRDs của dev/prod qua token backstage-agent (K8s tab, Flux tab)
  11. Bootstrap: ./deploy/setup.sh — 3 kind cluster + build image + helmfile (tier env → tokens → tier mgmt)

02Pipeline delivery & promotion

Image bất biến (sha-*) đi xuyên suốt: promote nghĩa là chuyển tag qua file env, không rebuild. dev tự động; prod qua PR có review.

03Image Automation + Promote đa vùng

Flux Image Automation trong cluster tự cập nhật tag, xác thực git bằng GitHub App duynhlab/gitops (App ID 4244512, scope Contents RW đúng 1 repo). staging tự động; prod-* qua PR review.

Điểm cốt lõi

  • Không PAT trong service repo: git-write chỉ nằm ở 1 GitHub App trong cluster, scope đúng repo gitops.
  • staging tự động, commit thẳng main; prod-* qua PR (push branch flux-image-* → GH Action mở PR → review).
  • Promote = retag, không rebuild: 0.2.0 → us-0.2.0 → eu-0.2.0, ImagePolicy mỗi vùng bắt prefix của mình.

Cần biết khi vận hành

  • Release theo git tag: push vX.Y.Z → CI build image X.Y.Z; ImagePolicy semver chọn bản cao nhất.
  • Thêm 2 controller image-reflector + image-automation vào FluxInstance.
  • Độ trễ ~1–2 phút theo scan interval.
  • PR review: staging bypass được vì CODEOWNERS không own apps/staging; prod-* buộc review.

06Trạng thái

  • E2E chạyĐã dựng thật 3 Kind cluster + toàn stack; Backstage lên :7007 (health 200), catalog thấy checkout, đủ 4 template. Flux + ESO + agent Ready trên dev & prod; GitHub App auth git OK; staging đã pull image 0.2.0.
  • Đã pushCả 3 repo đã commit/push, gồm các fix tìm ra lúc E2E: provider: github cho GitHub App auth, image CRD API v1 (không phải v1beta), helmfile wait:false+hook chờ FluxInstance Ready, secret githubapp idempotent.
  • Cần làmcheckout crashloop: image 0.2.0 là service thật (cần PostgreSQL/Temporal). Cho demo platform: đổi sang image stateless, hoặc cấp DB thật cho staging.
  • Cần làmTab CI/CD: thêm Callback URL + client secret cho GitHub App rồi redeploy với AUTH_GITHUB_CLIENT_ID/SECRET (xem deploy/README).

Tiếp tục ngày mai (state đã giữ nguyên)

  • Bật lại 3 cluster: docker start mgmt-control-plane dev-control-plane prod-control-plane (chờ ~1-2 phút).
  • Mở http://localhost:7007. Nếu IP cluster đổi sau restart, chạy lại token+mgmt tier (mục 4-5 deploy/setup.sh) để Backstage nối lại dev/prod.
  • Dựng lại từ đầu bất cứ lúc nào: ./deploy/setup.sh (đã có block export GitHub App trong deploy/README.md).

Pattern chung của mọi adopter

  • Bắt đầu từ Software Catalog — ai sở hữu gì, chạy ở đâu; giá trị thấy ngay từ ngày đầu.
  • Golden path templates — onboard service chuẩn hóa, dev không cần xin DevOps.
  • TechDocs — tài liệu sống cạnh code, render trong portal.
  • Plugin nội bộ là bước sau — Monzo viết plugin riêng khi catalog đã chín, không phải ngày đầu.

Map sang stack duynhlab

  • Catalog auto-discover từ gitops ✔ — như Epic/Unity.
  • Templates onboard/update-promote qua PR ✔ — chính là "golden path".
  • Flux tab + Kubernetes tab + CI/CD tab ✔ — hướng Monzo đã đi (deploy visibility trong portal).
  • Bước sau khi cần: TechDocs cho từng service, plugin đo chất lượng, scorecard.
